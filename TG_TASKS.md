# TG_TASKS — Telegram-бот (CPA / витрины)

**Этот файл — источник правды** по бэклогу, статусам задач и согласованному процессу. При **новом чате или смене контекста** агент заново читает `TG_TASKS.md` и продолжает с актуальных `[ ]` / `[x]`, не ожидая, что пользователь заново продиктует всё устно.

**Любой триггер на выполнение работы** (например: «сделай E1.3», «продолжим», «пофикси», «внеси правку», следующая подзадача того же эпика) означает **полный новый проход** цепочки ниже — без пропуска шагов и без перехода к коду до апрува плана. То же требование продублировано в **`AGENTS.md`**.

Легенда: `[ ]` не начато · `[~]` в работе · `[x]` готово · `[-]` отменено

После успешного коммита: в задаче `[ ]` → `[x]`, по желанию — хэш коммита в конце строки.

### Процесс выполнения задачи (строгий порядок)

1. **План** — кратко: шаги, паттерны, риски, варианты. **Без кода** в ответе.
2. **Апрув** — реализация только после явного согласия пользователя или после его правок к плану (новый короткий план при необходимости → снова апрув).
3. **Код** — правки в рабочей копии по согласованному плану.
4. **Ревью агентом** — отдельный проход ревью по диффу/изменениям; зафиксировать замечания; при необходимости исправить и снова пройти ревью до приемлемого состояния.
5. **Коммит** — только после шага 4; в сообщении указать **эпик и задачу** (например `E1.3` в начале или в теле сообщения однозначно).

Пуш в remote — сразу после коммита, если это часть принятого процесса в ветке, либо по договорённости; цепочка **заканчивается коммитом** как точкой фиксации результата ревью.

В `AGENTS.md` — краткая отсылка сюда, чтобы правило подхватывалось между сессиями.

## Epic E1 — TelegramDelivery (webhook + безопасность)

- E1.1 HTTP POST webhook: путь, проверка `secret_token` или path secret — `POST /telegram/webhook` и `POST /telegram/webhook/:pathSecret`; env: `TELEGRAM_WEBHOOK_SECRET_TOKEN` (заголовок из `telegram.webhookSecretHeaderName`, по умолчанию `x-telegram-bot-api-secret-token`; переопределение — `TELEGRAM_WEBHOOK_SECRET_HEADER_NAME`), опционально `TELEGRAM_WEBHOOK_PATH_SECRET`; в `production` без секретов — `401`
- [x] E1.2 Быстрый ACK: 200 сразу, тело update в очередь (BullMQ) — очередь `telegramIncoming`, job `telegram-webhook-update`, payload `{ raw }`; после успешной авторизации `await` постановки в Redis (быстрый round-trip)
- [x] E1.3 Лимит тела, таймаут, логи без PII — глобальный JSON `10mb` в `main.ts` (`bodyParser: false` + `useBodyParser`); на webhook интерцептор: `telegram.webhookMaxBodyBytes` (дефолт 512KiB, env `TELEGRAM_WEBHOOK_MAX_BODY_BYTES`), при превышении **413**; `telegram.webhookRequestTimeoutMs` / `TELEGRAM_WEBHOOK_REQUEST_TIMEOUT_MS` (`0` = выкл.) — `socket.setTimeout` на время обработки; `summarizeTelegramUpdateForLog` + `TELEGRAM_WEBHOOK_LOG_SUMMARY=true` — только `update_id` и типы апдейта, без текста сообщений

## Epic E2 — Дедупликация и идемпотентность

- [x] E2.1 Дедуп по `update_id` (Redis `SET` NX + TTL) — ключ `telegram:webhook:dedup:v1:{TELEGRAM_BOT_ID|default}:{update_id}`, TTL `TELEGRAM_WEBHOOK_DEDUPE_TTL_SECONDS` (60..604800 с, дефолт 86400); перед Bull `claimFirstDelivery`; дубликат → не ставим job, **200**; без `update_id` / ошибка Redis → **fail-open** (ставим job)

- E2.3 Идempotency-key для дорогих действий (запуск обхода в окне)

## Epic E10 — Настройки платформы (БД) и режим LLM / kill-switch

Цель: два равноправных пути (**UI** и **LLM-координатор**), но в Mongo — **настройки**, позволяющие мгновенно отключить LLM для всех или оставить только платным; функциональность бота **не теряется** (остается E7.4).

- E10.1 Коллекция настроек (например `platform_settings`): документ singleton (`_id: 'global'` или один фиксированный ключ). Поля минимум: `llmCoordinatorPolicy`: `on`  `off`  `paid_subscribers_only` (имена зафиксировать в коде + zod). Опционально: `updatedAt`, `updatedBy` для аудита
- E10.2 Репозиторий + zod при upsert; дефолт при отсутствии документа (например `on` в dev, в prod — явная инициализация миграцией/скриптом)
- E10.3 Сервис разрешений: `canUseLlm(user)` = `llmCoordinatorPolicy` из БД **и** tier подписки (E8.5): `off` → всегда false; `on` → true (если не уперлись в E5); `paid_subscribers_only` → true только для paid (trial/free → false)
- E10.4 **Быстрое применение**: кэш настроек с коротким TTL (5–30 с) и/или инвалидация по записи; без обязательного редеплоя для «критового» отключения
- E10.5 Входящий свободный текст: если `canUseLlm` === false — **не** вызывать OpenRouter; ответ шаблоном («сейчас настройками отключён диалоговый режим, используйте меню ниже») + главная клавиатура (E7.1)
- E10.6 Способ смены политики операционно: защищённый HTTP-эндпоинт, Nest CLI-команда или админ-скрипт (секрет/роль); смена одного поля в документе достаточна для kill-switch
- E10.7 Зафиксировать матрицу «политика × tier × лимит E5» в этом файле или в коде-комментарии к `canUseLlm` (чтобы не плодить расхождений)

## Критерий MVP

Webhook → очередь → текст + callback; dedup `update_id`; outbound с лимитом и 429; привязка TG user; 1 витрина + 1 метка; запуск по кнопке + короткий отчёт; LLM за флагом, без LLM — меню. Плюс: `**llmCoordinatorPolicy` в БД**, режим `off` или `paid_subscribers_only` проверен вручную: бот остаётся полезным только через UI.

## Приложение A — User journey (E0.1)

### Акторы

- **Рекламодатель / CPA**: владелец обходов, платит за подписку после trial.
- **Бот**: адаптер Telegram + координатор (LLM) + вызовы домена (витрины, расписание, отчёты).

### Шаг 1 — Вход и онбординг

1. Пользователь жмёт ссылку на бота или `/start`.
2. Бот кратко объясняет ценность: «фиксируете витрины и метки — бот проверяет, на месте ли рефка, присылает отчёт по расписанию или по кнопке».
3. Показывается статус **trial / подписка** (если paywall уже включён) и одна главная кнопка «Начать» или сразу главное меню.

### Шаг 2 — Привязка аккаунта

1. Создаётся или находится внутренний **User** по `telegram_id` (без лишних PII в промпте).
2. Дальнейшие действия привязаны к этому пользователю и его лимитам (LLM, обходы).

### Шаг 3 — Витрины (ссылки)

1. Пользователь добавляет витрины **диалогом** («добавь https://…») и/или через меню «Мои витрины → Добавить».
2. Бот валидирует URL на стороне приложения (схема, длина, denylist), сохраняет в доменную сущность чеклиста/витрин.
3. Список витрин можно посмотреть и удалить пунктом или пачкой с подтверждением (см. E0.3).

### Шаг 4 — Метки оферов

1. Пользователь задаёт **метки** (например `sravni`, `startracking`, `google` или произвольные строки), которые используются при разборе страницы / сопоставлении с офферами.
2. Метки отображаются в настройках и участвуют в каждом обходе по умолчанию (или с явным выбором «для этого запуска» — на усмотрение MVP).

### Шаг 5 — Расписание

1. Один **глобальный** сценарий расписания на все обходы пользователя: например «каждый день в 10:00» или «пн/ср/пт 09:00» (крон или предустановки в UI).
2. Изменение расписания — через диалог LLM или форму/кнопки; валидируется на бэкенде.

### Шаг 6 — Запуск и отчёт

1. **По кнопке** «Проверить сейчас» ставится задача в существующий pipeline (Playwright / dispatch), без блокировки webhook.
2. **По расписанию** тот же pipeline запускается для всех закреплённых витрин и меток.
3. По завершении пользователь получает **отчёт** в Telegram (кратко в сообщении + при необходимости файл/несколько сообщений с учётом лимитов TG).

### Шаг 7 — Подписка

1. По окончании trial бот предлагает оплату; без активной подписки режутся дорогие действия (обходы, LLM) согласно политике E8.

### Альтернативные потоки

- **Только кнопки**: пользователь не пишет свободным текстом — всё через меню; LLM не вызывается.
- **Глобально или по тарифу выключен LLM** (E10): свободный текст не уходит в модель; пользователь видит подсказку и то же меню — сценарии витрин / меток / расписания / отчётов как в E7.4.
- **Ошибка LLM / лимит**: короткое сообщение + переход в меню; команды не выполняются частично без явного подтверждения для деструктивных операций.

## Приложение B — Системный промпт LLM (E0.2)

Ниже — **черновик системной роли** для координатора (OpenRouter). Ответ модели парсится на бэкенде через **zod**; при невалидном JSON — fallback без выполнения команд.

### Роль

Ты парсер намерений пользователя CPA/рекламного бота. Ты **не** выполняешь HTTP-запросы, **не** открываешь ссылки и **не** придумываешь URL, e-mail, токены и платежи. Ты только предлагаешь **команды** для бэкенда.

### Формат ответа (строго)

Верни **один** JSON-объект без Markdown и без текста вокруг:

```json
{
  "commands": [ ... ],
  "assistant_message": "краткий текст пользователю на русском, 1-3 предложения"
}
```

- `commands` — массив из **0..10** элементов.
- Если данных не хватает — **одна** команда типа `ASK_CLARIFICATION` с полем `question`.
- Если пользователь просто здоровается — `commands: []` и дружелюбный `assistant_message` с подсказкой меню.

### Типы команд (`type`)

Каждая команда: `{ "type": "...", ...поля }`.


| `type`                | Поля                                                                                             | Заметки                                                                                 |
| --------------------- | ------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------- |
| `ADD_VITRINA_URL`     | `url` (string)                                                                                   | Только URL, явно написанный пользователем. Если URL нет в тексте — `ASK_CLARIFICATION`. |
| `REMOVE_VITRINA`      | `url` или `index` (number)                                                                       | Если неоднозначно — `ASK_CLARIFICATION`.                                                |
| `LIST_VITRINAS`       | —                                                                                                |                                                                                         |
| `SET_OFFER_TAGS`      | `tags: string[]`                                                                                 | Нормализуй к нижнему регистру, без дубликатов, max 20 тегов, длина тега ≤ 64.           |
| `LIST_OFFER_TAGS`     | —                                                                                                |                                                                                         |
| `SET_SCHEDULE`        | `cron` (string) **или** `preset` (`daily_10`, `weekdays_9`, … — только если бэкенд поддерживает) | Если расписание неоднозначно — `ASK_CLARIFICATION`.                                     |
| `CLEAR_SCHEDULE`      | —                                                                                                | Деструктивно → бэкенд может потребовать подтверждение.                                  |
| `RUN_CHECK_NOW`       | —                                                                                                | Дорогая операция → бэкенд может потребовать подтверждение.                              |
| `GET_LAST_REPORT`     | —                                                                                                |                                                                                         |
| `SUBSCRIPTION_STATUS` | —                                                                                                |                                                                                         |
| `ASK_CLARIFICATION`   | `question` (string)                                                                              | Один конкретный вопрос.                                                                 |


Запрещено: выдумывать `url`, `cron`, списки витрин «с потолка». Разрешено: уточнить у пользователя.

### Пример

Вход: «добавь витрину [https://example.com/path](https://example.com/path) и метки Sravni, Google».

```json
{
  "commands": [
    { "type": "ADD_VITRINA_URL", "url": "https://example.com/path" },
    { "type": "SET_OFFER_TAGS", "tags": ["sravni", "google"] }
  ],
  "assistant_message": "Добавлю витрину и метки sravni, google. Если нужно другое написание меток — напишите."
}
```

### Язык

`assistant_message` и `ASK_CLARIFICATION.question` — **русский**, кратко.

## Приложение C — Политика подтверждений (E0.3)

Цель: не выполнять деструктивные или дорогие действия **только по одному сообщению LLM** без согласия пользователя в Telegram.

### Всегда без дополнительного inline-подтверждения (MVP)

После валидации команд на бэкенде:

- `LIST_VITRINAS`, `LIST_OFFER_TAGS`, `SUBSCRIPTION_STATUS`, `GET_LAST_REPORT` (только чтение).
- `ADD_VITRINA_URL` — если URL прошёл валидацию и **явно** присутствует в тексте пользователя (не «догадка» модели).
- `SET_OFFER_TAGS` — если список не пустой и все теги из ответа **цитируют** слова из сообщения пользователя (иначе → уточнение или подтверждение).

### Inline «Подтвердить» / «Отмена» (один message + callback)

Показываем краткое резюме действия и две кнопки:


| Команда(ы)                                           | Причина                                                                                 |
| ---------------------------------------------------- | --------------------------------------------------------------------------------------- |
| `REMOVE_VITRINA`                                     | Необратимо                                                                              |
| `CLEAR_SCHEDULE`                                     | Обнуляет автоматические обходы                                                          |
| `SET_SCHEDULE`                                       | Если меняет уже существующее расписание (не первичная настройка)                        |
| `RUN_CHECK_NOW`                                      | Дорого (Playwright / очередь); опционально разрешить без подтверждения для платных tier |
| Пакет: **≥2** деструктивных или смешанных add+remove | Снижает риск неверной интерпретации                                                     |


Идемпотентность: callback несёт `confirmation_id` (UUID в Redis с TTL 15 мин), повторный клик не дублирует действие.

### После отклонения

«Отмена» — бот отвечает нейтрально, состояние черновика команд сбрасывается; пользователь может повторить формулировку или воспользоваться меню.

### Без LLM

Те же правила для действий из **чистого UI** (кнопки меню): удаление витрины и сброс расписания всегда с подтверждением.
