Webhook → очередь → текст + callback; dedup `update_id`; outbound с лимитом и 429; привязка TG user; 1 витрина + 1 метка; запуск по кнопке + короткий отчёт; LLM за флагом, без LLM — меню.

## Приложение A — User journey (E0.1)

### Акторы

- **Рекламодатель / CPA**: владелец обходов, платит за подписку после trial.
- **Бот**: адаптер Telegram + координатор (LLM) + вызовы домена (витрины, расписание, отчёты).

### Шаг 1 — Вход и онбординг

1. Пользователь жмёт ссылку на бота или `/start`.
2. Бот кратко объясняет ценность: «фиксируете витрины и метки — бот проверяет, на месте ли рефка, присылает отчёт по расписанию или по кнопке».
3. Показывается статус **trial / подписка** (если paywall уже включён) и одна главная кнопка «Начать» или сразу главное меню.

### Шаг 2 — Привязка аккаунта

4. Создаётся или находится внутренний **User** по `telegram_id` (без лишних PII в промпте).
5. Дальнейшие действия привязаны к этому пользователю и его лимитам (LLM, обходы).

### Шаг 3 — Витрины (ссылки)

6. Пользователь добавляет витрины **диалогом** («добавь https://…») и/или через меню «Мои витрины → Добавить».
7. Бот валидирует URL на стороне приложения (схема, длина, denylist), сохраняет в доменную сущность чеклиста/витрин.
8. Список витрин можно посмотреть и удалить пунктом или пачкой с подтверждением (см. E0.3).

### Шаг 4 — Метки оферов

9. Пользователь задаёт **метки** (например `sravni`, `startracking`, `google` или произвольные строки), которые используются при разборе страницы / сопоставлении с офферами.
10. Метки отображаются в настройках и участвуют в каждом обходе по умолчанию (или с явным выбором «для этого запуска» — на усмотрение MVP).

### Шаг 5 — Расписание

11. Один **глобальный** сценарий расписания на все обходы пользователя: например «каждый день в 10:00» или «пн/ср/пт 09:00» (крон или предустановки в UI).
12. Изменение расписания — через диалог LLM или форму/кнопки; валидируется на бэкенде.

### Шаг 6 — Запуск и отчёт

13. **По кнопке** «Проверить сейчас» ставится задача в существующий pipeline (Playwright / dispatch), без блокировки webhook.
14. **По расписанию** тот же pipeline запускается для всех закреплённых витрин и меток.
15. По завершении пользователь получает **отчёт** в Telegram (кратко в сообщении + при необходимости файл/несколько сообщений с учётом лимитов TG).

### Шаг 7 — Подписка

16. По окончании trial бот предлагает оплату; без активной подписки режутся дорогие действия (обходы, LLM) согласно политике E8.

### Альтернативные потоки

- **Только кнопки**: пользователь не пишет свободным текстом — всё через меню; LLM не вызывается.
- **Ошибка LLM / лимит**: короткое сообщение + переход в меню; команды не выполняются частично без явного подтверждения для деструктивных операций.
